# v0.3 Implementation Plan — Admin Panel & Platform Features

**Branch:** `feature/v0.3-admin-platform`
**Target version:** `0.3.0`
**Status:** In progress

---

## Overview

This plan covers five interconnected feature areas delivered as a single minor version. All
features are **opt-in by default** (feature flag policy unchanged). The admin panel is the
connective tissue: it manages users, invites, API keys, account locks, and can override any
config value that makes sense to control at runtime.

**Priority order:** admin panel foundation → progressive lockout → invite system → M2M API
keys → user profile → admin config overrides.

---

## Feature 1 — Admin Panel

### Purpose
A first-party authenticated admin area served from the package, accessible at a configurable
path (default `/admin`). Admin access is controlled by a `role` field on the user record.
The panel surfaces controls for all other v0.3 features and exposes read-only health/audit
views.

### Sections

| Section | What it shows / does |
|---|---|
| **Users** | List all users, filter by status/role; view profile; change role; suspend/reactivate |
| **Waitlist** | Pending users; approve or reject with optional reason email |
| **Invites** | All invite codes; who created them, how many uses remain, expiry; revoke |
| **Account locks** | Locked/frozen accounts; unlock manually; view attempt history |
| **API keys** | Create, rotate, revoke M2M keys; set scopes and expiry |
| **Config** | Override runtime config values stored in DB (see Feature 5) |
| **Audit log** | Read-only feed from the existing audit module |

### Auth model
- New `role` column on `users`: `"user" | "admin"`. Default `"user"`.
- `createSecureAuth` accepts `admin.enabled` (default `false`) and `admin.path` (default
  `"/admin"`).
- Middleware extension: `createSecureAuthMiddleware` blocks non-admin requests to the admin
  path automatically when `admin.enabled = true`.
- First admin: seeded via `ADMIN_BOOTSTRAP_EMAIL` env var — on first server start, if the
  user exists and has `role = "user"`, it is promoted to `"admin"`. This is a one-time
  bootstrap; the env var has no effect once at least one admin exists.

## Feature 2 — Progressive Account Lockout

### Purpose
Replace the current flat rate limit with a graduated response: warnings → temporary freeze
→ permanent lock. Configurable per threshold and duration. Admin can unlock.

### Behaviour

| Failed attempts | Default action | Configurable? |
|---|---|---|
| 3 | 5-minute freeze | yes |
| 6 | 30-minute freeze | yes |
| 9 | 4-hour freeze | yes |
| 12 | Permanent lock (requires admin unlock) | yes |

During a freeze, the credentials login handler returns `"account_frozen"` with a
`retryAfterSeconds` value. The `LoginPage` component renders a countdown and blocks
the submit button.

After the permanent lock threshold is reached, the handler returns `"account_locked"` and
the page shows a message to contact support. The admin panel shows the account under
**Account locks** and provides an unlock button.

### Config shape
```ts
auth: {
  accountLockout?: {
    enabled: boolean;               // default: false (opt-in)
    thresholds?: Array<{
      attempts: number;             // cumulative failed attempts
      action: "freeze" | "lock";
      freezeDurationSeconds?: number; // required when action = "freeze"
    }>;
    // Default thresholds (when enabled but thresholds not provided):
    // [
    //   { attempts: 3,  action: "freeze", freezeDurationSeconds: 300 },
    //   { attempts: 6,  action: "freeze", freezeDurationSeconds: 1800 },
    //   { attempts: 9,  action: "freeze", freezeDurationSeconds: 14400 },
    //   { attempts: 12, action: "lock" },
    // ]
  };
};
```

### Schema additions
```sql
CREATE TABLE login_attempt_counters (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID REFERENCES users(id) ON DELETE CASCADE,
  email       TEXT,                 -- for pre-registration attempts
  attempts    INTEGER NOT NULL DEFAULT 0,
  frozen_until TIMESTAMPTZ,
  locked_at   TIMESTAMPTZ,
  last_attempt_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  unlocked_at TIMESTAMPTZ,         -- set by admin unlock
  unlocked_by UUID REFERENCES users(id),
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE UNIQUE INDEX ON login_attempt_counters (user_id) WHERE user_id IS NOT NULL;
CREATE UNIQUE INDEX ON login_attempt_counters (email) WHERE user_id IS NULL;
```

### Env vars

## Feature 3 — Invite System

### Purpose
Gate new account creation behind an approval queue and/or invite codes. Two modes:

- **Waitlist mode:** anyone can register, but new accounts start with `status = "pending"`.
  They can log in but see a "pending approval" screen instead of the dashboard. Admin
  approves them, which sets `status = "active"` and sends an approval email.
- **Invite-only mode:** registration is blocked unless the request carries a valid invite
  code (URL param `?code=…` or form input). Invite codes are generated by approved users
  or admins.
- **Both modes together:** registration requires a code AND the new account starts pending
  (double-gate). Useful for very closed betas.

### Invite code lifecycle
1. Approved user or admin generates a code (admin sets quota per user or unlimited).
2. Code carries: creator ID, max uses, expiry date, optional email restriction.
3. Recipient registers with the code in the URL (`/register?code=ABC123`) or types it in
   the registration form.
4. On successful registration the code's `usedCount` increments. When `usedCount = maxUses`
   the code becomes inactive.
5. If waitlist mode is also active, the invited user still starts as `pending`.

### Config shape
```ts
invites?: {
  enabled: boolean;               // default: false
  requireApproval: boolean;       // waitlist gate — default: false
  requireInviteCode: boolean;     // code gate — default: false
  defaultQuotaPerUser: number;    // how many codes each approved user gets — default: 0
  codeExpiryDays: number;         // default: 30
  adminUnlimitedQuota: boolean;   // admins bypass quota — default: true
};
```

### Schema additions
```sql
CREATE TABLE invite_codes (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  code         TEXT NOT NULL UNIQUE,
  created_by   UUID REFERENCES users(id) ON DELETE SET NULL,
  max_uses     INTEGER,           -- NULL = unlimited (admin only)
  used_count   INTEGER NOT NULL DEFAULT 0,
  email_hint   TEXT,              -- optional: code only valid for this email
  expires_at   TIMESTAMPTZ,
  revoked_at   TIMESTAMPTZ,
  revoked_by   UUID REFERENCES users(id),
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE UNIQUE INDEX ON invite_codes (code);

CREATE TABLE invite_uses (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  code_id      UUID NOT NULL REFERENCES invite_codes(id),
  used_by      UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  used_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### Package exports
```ts

// Shown after registration when status = "pending"; blocks dashboard access.

// New UI path

```

## Feature 4 — Machine-to-Machine Auth (API Keys)

### Purpose
Allow server-side clients (other services, CI pipelines, CLIs) to authenticate against APIs
that the consumer exposes, using static API keys managed by the package. The package
validates the key, resolves the principal, and makes it available in the request context.
No session cookies are involved.

### Flow
1. Admin creates an API key in the admin panel (or via `POST /api/auth/admin/api-keys`).
2. The key is shown once at creation time (stored as bcrypt hash).
3. Client sends `Authorization: Bearer <key>` on every request.
4. The consumer calls `secureAuth.middleware.validateApiKey(request)` (or uses the
   provided Next.js route handler helper) to authenticate the request.
5. The resolved principal carries: `keyId`, `name`, `scopes`, `createdBy`, `expiresAt`.

### Config shape
```ts
apiKeys?: {
  enabled: boolean;               // default: false
  defaultExpiryDays?: number;     // default: 365, null = never
};
```

### Schema additions
```sql
CREATE TABLE api_keys (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name         TEXT NOT NULL,
  key_hash     TEXT NOT NULL UNIQUE,   -- bcrypt hash of the raw key
  key_prefix   TEXT NOT NULL,          -- first 8 chars, shown in list for identification
  scopes       TEXT[] NOT NULL DEFAULT '{}',
  created_by   UUID REFERENCES users(id) ON DELETE SET NULL,
  last_used_at TIMESTAMPTZ,
  expires_at   TIMESTAMPTZ,
  revoked_at   TIMESTAMPTZ,
  revoked_by   UUID REFERENCES users(id),
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### Package exports
```ts
// Helper for consumers to protect their own API routes
export { withApiKeyAuth } from "@tgoliveira/secure-auth/next";
// Usage: export const GET = withApiKeyAuth(secureAuth, handler, { scopes: ["read:data"] });

export type { ApiKeyPrincipal } from "@tgoliveira/secure-auth";
```

### Env vars
```env

AUTH_API_KEYS_DEFAULT_EXPIRY_DAYS=365
```

## Feature 5 — User Profile

### Purpose
Optional user profile layer: display name, avatar URL, bio. When enabled via config, the
package pulls available fields from OAuth providers on login (name, picture) and stores
them. Users can edit their profile in the account settings area.

### Config shape
```ts
profile?: {
  enabled: boolean;               // default: false
  allowAvatarUpload?: boolean;    // default: false (URL-only when false)
};
```

### Schema additions
```sql
ALTER TABLE users ADD COLUMN display_name TEXT;
ALTER TABLE users ADD COLUMN avatar_url TEXT;
ALTER TABLE users ADD COLUMN bio TEXT;
ALTER TABLE users ADD COLUMN profile_updated_at TIMESTAMPTZ;
```

### OAuth sync
On every successful OAuth login, if `profile.enabled = true`, the package updates
`display_name` and `avatar_url` from the provider token (only if the user has not
manually edited their profile — tracked via `profile_updated_at`).

Fields synced by provider:

| Provider | display_name | avatar_url |
|---|---|---|
| Google | `name` | `picture` |
| GitHub | `name` | `avatar_url` |
| Apple | `name.firstName + lastName` | — |
| Microsoft | `name` | — |

### Package exports
```ts
// Shown in account settings when profile.enabled
// ProfileSettingsPage is included in AccountSettingsPage when enabled;
// consumers do not need to add a separate route.
export { ProfileSettingsPage } from "@tgoliveira/secure-auth/react";
```

### Env vars
```env
AUTH_PROFILE_ENABLED=false
AUTH_PROFILE_ALLOW_AVATAR_UPLOAD=false
```

## Feature 6 — Admin Config Overrides

### Purpose
Allow runtime config values to be changed from the admin panel without redeploying. The
priority chain becomes: **admin DB override → env var → package default**.

Only a curated subset of config values are overridable at runtime — those that are safe to
change without a restart and that have clear operational value (e.g., feature flags, lockout
thresholds, session durations). Secrets (NextAuth secret, encryption keys) and structural
config (database URL, OAuth credentials) are never overridable from the admin panel.

### Overridable values (initial set)

| Config path | Admin UI label |
|---|---|
| `auth.accountLockout.enabled` | Account lockout on/off |
| `auth.accountLockout.thresholds` | Lockout thresholds (JSON editor) |
| `invites.enabled` | Invite system on/off |
| `invites.requireApproval` | Require waitlist approval |
| `invites.requireInviteCode` | Require invite code |
| `invites.defaultQuotaPerUser` | Default invite quota |
| `invites.codeExpiryDays` | Invite code expiry |
| `passwordPolicy.enforcement` | Password enforcement level |
| `passwordPolicy.minLength` | Minimum password length |
| `passwordPolicy.checkBreachedPasswords` | HIBP check |
| `sessions.maxAgeSeconds` | Session lifetime |
| `sessions.singleActiveSession` | Single-session mode |
| `auth.magicLink.enabled` | Magic link login |
| `auth.securityNotifications.enabled` | Security notification emails |
| `profile.enabled` | User profile |
| `apiKeys.enabled` | API keys |

### Schema additions
```sql
CREATE TABLE admin_config_overrides (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  key        TEXT NOT NULL UNIQUE,   -- dot-notation path, e.g. "sessions.maxAgeSeconds"
  value      JSONB NOT NULL,
  updated_by UUID REFERENCES users(id) ON DELETE SET NULL,
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### Runtime behaviour
`createSecureAuth` returns a `config` accessor that is re-evaluated on each request
(not frozen at startup). For the DB override path, the package caches the override
table in memory with a configurable TTL (default 60 seconds), so there is no DB round-trip
per request.

## Implementation Phases

Each phase is a self-contained PR to `feature/v0.3-admin-platform`. Tests must pass before
merging each phase into the branch.

### Phase 1 — Foundation (schema + admin auth + user list)
- Add `role` and `status` to `users` schema
- `ADMIN_BOOTSTRAP_EMAIL` seeding logic
- Admin middleware guard in `createSecureAuthMiddleware`
- `AdminUsersPage`: list, filter, change role, suspend/reactivate
- Admin API routes: `GET /api/auth/admin/users`, `PATCH /api/auth/admin/users/:id`
- consumer-demo: add `/admin/*` pages
- Tests: admin guard, user list API, role change, bootstrap

### Phase 2 — Progressive Lockout
- `login_attempt_counters` schema
- `lockoutService`: record attempt, compute state (ok/frozen/locked), unlock
- Integrate into credentials login handler
- `LoginPage` UI: frozen countdown, locked message
- Admin: `AdminLocksPage`, unlock API
- consumer-demo: `AdminLocksPage` wired
- Tests: threshold progression, freeze expiry, admin unlock

### Phase 3 — Invite System
- `invite_codes` + `invite_uses` schema
- `inviteService`: generate, validate, consume, revoke
- `RegisterPage` extension: code field + URL param support
- `WaitlistPendingPage`: shown when `status = "pending"`
- Middleware: redirect `status = "pending"` users to waitlist page
- Admin: `AdminWaitlistPage`, `AdminInvitesPage`
- Approval email via existing notification service
- consumer-demo: waitlist + invite pages wired
- Tests: code validation, quota, expiry, waitlist redirect, approval email

### Phase 4 — API Keys (M2M)
- `api_keys` schema
- `apiKeyService`: create (returns raw key once), validate (bcrypt check), revoke
- `withApiKeyAuth` Next.js helper
- Admin: `AdminApiKeysPage`
- consumer-demo: admin page wired + example protected route
- Tests: key creation, hash validation, revocation, expired key rejection

### Phase 5 — User Profile
- Schema: `display_name`, `avatar_url`, `bio`, `profile_updated_at` on `users`
- OAuth sync hook in `auth-options.ts` (jwt/session callbacks)
- `ProfileSettingsPage` embedded in `AccountSettingsPage` when `profile.enabled`
- Profile read API: `GET /api/auth/account/profile`
- Profile update API: `PATCH /api/auth/account/profile`
- Tests: OAuth sync, manual edit blocks sync, profile update validation

### Phase 6 — Admin Config Overrides
- `admin_config_overrides` schema
- Config cache layer with 60s TTL
- Priority resolver: override → env → default
- `AdminConfigPage`: table of overridable keys with current source and value
- Tests: override takes priority, TTL invalidation, non-overridable keys rejected

## Open Questions / Decisions

1. **Admin UI framework**: Use the existing `Button`, `Card`, `Input`, `Alert` primitives
   from the package. Tables and data grids are new primitives to add — simple `<table>`
   wrappers styled with CSS variables, no external dependency.
2. **Avatar upload** (Phase 5, when `allowAvatarUpload = true`): Storage is consumer-
   provided. The package accepts a `profile.uploadHandler` callback and calls it with the
   raw file buffer; the consumer returns a URL. This keeps cloud provider dependencies out
   of the package.
3. **Scopes for API keys**: v0.3 ships with free-form string scopes. RBAC (planned for a
   future version) will formalize scope definitions and permissions.
4. **Admin config cache TTL**: Default 60 seconds. Configurable via
   `AUTH_ADMIN_CONFIG_CACHE_TTL_SECONDS`. Set to `0` to disable caching (DB hit per
   request — only for development).
