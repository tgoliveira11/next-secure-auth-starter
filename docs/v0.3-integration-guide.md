# v0.3 Integration Guide

**Package version:** `0.3.0`
**Audience:** Agents and engineers adding v0.3 features to an existing consumer application that already integrates `@tgoliveira/secure-auth`.

This guide is a step-by-step checklist. Each feature is independent and opt-in. You can integrate them in any order. Skip sections that do not apply.

**Prerequisites:** The application must already have a working v0.2 integration. If not, complete [consumer-quick-start.md](./consumer-quick-start.md) first.

---

## Quick reference — what v0.3 adds

| Feature | Config key | Env var | API routes added | UI pages added |
|---|---|---|---|---|
| Admin panel | `admin.enabled` | `AUTH_ADMIN_ENABLED` | `adminUsers`, `adminWaitlist`, `adminInvites`, `adminLocks`, `adminApiKeys`, `adminConfig` | `AdminUsersPage`, `AdminWaitlistPage`, `AdminInvitesPage`, `AdminLocksPage`, `AdminApiKeysPage`, `AdminConfigPage` |
| Account lockout | `accountLockout.enabled` | `AUTH_ACCOUNT_LOCKOUT_ENABLED` | none (internal) | `AdminLocksPage` |
| Invite / waitlist | `invites.enabled` | `AUTH_INVITES_ENABLED` | `adminWaitlist`, `adminInvites` | `AdminWaitlistPage`, `AdminInvitesPage`, `WaitlistPendingPage` |
| API keys (M2M) | `apiKeys.enabled` | `AUTH_API_KEYS_ENABLED` | `adminApiKeys` | `AdminApiKeysPage` |
| User profile | `profile.enabled` | `AUTH_PROFILE_ENABLED` | `accountProfile` | (embed in your profile UI) |
| Config overrides | (always available when `admin.enabled`) | — | `adminConfig` | `AdminConfigPage` |

## Step 1 — Upgrade the package

```bash
npm install @tgoliveira/secure-auth@0.3.0
```

After upgrading, run the new database migrations before starting the server:

```bash
npx drizzle-kit migrate
```

> **What migration `0002_v0_3_admin_platform` adds:** `login_attempt_counters`, `invite_codes`, `invite_uses`, `api_keys`, `admin_config_overrides`, and new columns on `users` (`role`, `status`, `display_name`, `avatar_url`, `bio`, `profile_updated_at`).

---

## Step 2 — Extend `createSecureAuth` config

Open the file where you call `createSecureAuth(config)` (e.g. `src/lib/secure-auth.ts`) and add the new config sections:

```typescript
export const secureAuth = createSecureAuth({
  // ... existing config unchanged ...

  // v0.3 additions — all opt-in, all default false/0
  admin: {
    enabled: true,                        // required to use any admin UI
    path: "/admin",                       // customize if needed
    bootstrapEmail: process.env.ADMIN_BOOTSTRAP_EMAIL,  // first admin
    configCacheTtlSeconds: 60,            // in-memory config override TTL
  },
  accountLockout: {
    enabled: true,
    // thresholds are optional — defaults: freeze at 3/6/9 attempts, lock at 12
    thresholds: [
      { attempts: 3,  action: "freeze", freezeDurationSeconds: 300   },
      { attempts: 6,  action: "freeze", freezeDurationSeconds: 1800  },
      { attempts: 9,  action: "freeze", freezeDurationSeconds: 14400 },
      { attempts: 12, action: "lock" },
    ],
  },
  invites: {
    enabled: true,
    requireApproval: false,   // true = all new accounts start as "pending"
    requireInviteCode: false, // true = registration blocked without a code
    defaultQuotaPerUser: 5,   // invite codes each user can generate
    codeExpiryDays: 30,
  },
  apiKeys: {
    enabled: true,
  },
  profile: {
    enabled: true,
  },
});
```

If you use a `buildSecureAuthConfigFromEnv()` helper (recommended), add the corresponding env var reads instead of hardcoding values — see Step 3.

## Step 3 — Add environment variables

Add to your `.env.local` (and `.env.example`):

```bash
# Admin panel
AUTH_ADMIN_ENABLED=true
AUTH_ADMIN_PATH=/admin
ADMIN_BOOTSTRAP_EMAIL=your-admin@example.com   # promote on first start; remove after

# Progressive account lockout
AUTH_ACCOUNT_LOCKOUT_ENABLED=true

# Invite / waitlist
AUTH_INVITES_ENABLED=true
AUTH_INVITES_REQUIRE_APPROVAL=false   # set true to enable waitlist flow
AUTH_INVITES_REQUIRE_CODE=false       # set true to require an invite code at registration
AUTH_INVITES_DEFAULT_QUOTA=5
AUTH_INVITES_CODE_EXPIRY_DAYS=30

# API keys (machine-to-machine)
AUTH_API_KEYS_ENABLED=true

# User profile
AUTH_PROFILE_ENABLED=true
```

If you have a `buildSecureAuthConfigFromEnv` helper, update it to read these into the config shape shown in Step 2. Use the pattern from [`apps/consumer-demo/src/lib/env/secure-auth-from-env.ts`](../apps/consumer-demo/src/lib/env/secure-auth-from-env.ts).

## Step 6 — Protect the admin section with middleware

Open `src/middleware.ts` and ensure the admin path is covered by `createSecureAuthMiddleware`. The middleware already blocks non-admin access when `admin.enabled = true` — you only need to make sure the path is inside the matcher:

```typescript
import { createSecureAuthMiddleware, defaultSecureAuthMiddlewareMatcher } from "@tgoliveira/secure-auth";
import { secureAuth } from "@/lib/secure-auth";

export const middleware = createSecureAuthMiddleware(secureAuth);

export const config = {
  matcher: defaultSecureAuthMiddlewareMatcher,
  // or a custom array — ensure "/admin/:path*" and "/api/auth/admin/:path*" are included
};
```

If you use a custom matcher, add these patterns:

```
"/admin/:path*",
"/api/auth/admin/:path*",
```

## Step 7 — Bootstrap the first admin

On first deploy, set `ADMIN_BOOTSTRAP_EMAIL` to the email of the first admin user. On the next server start, if that user exists and has `role = "user"`, they are promoted to `"admin"` automatically. This promotion runs once; after at least one admin exists the env var has no further effect. Remove the env var from production after bootstrap.

Alternatively, run a one-shot SQL query:

```sql
UPDATE users SET role = 'admin' WHERE email = 'your@email.com';
```

## Step 8 — Use API keys for machine-to-machine auth (optional)

If you have internal services or webhooks that need to call your app's API without a user session, use `withApiKeyAuth`:

**`src/app/api/internal/data/route.ts`**
```typescript
import { withApiKeyAuth } from "@tgoliveira/secure-auth";
import { secureAuth } from "@/lib/secure-auth";
import type { NextRequest } from "next/server";

export const GET = withApiKeyAuth(
  secureAuth.services,
  async (request: NextRequest, principal) => {
    // principal: { keyId, name, scopes, createdBy, expiresAt }
    return Response.json({ ok: true, caller: principal.name });
  },
  { scopes: ["data:read"] }   // optional — enforces scope on the key
);
```

Callers authenticate with an `Authorization: Bearer <rawKey>` header. Raw keys are shown once at creation in the `AdminApiKeysPage`.

---

## Step 9 — Embed profile editing (optional)

The profile feature provides `GET /api/account/profile` and `POST /api/account/profile`. Fields: `displayName`, `avatarUrl`, `bio`. Render a profile form in your existing account settings page and call these endpoints directly, or embed the API calls from a client component.

OAuth logins automatically sync `name` and `image` into `displayName`/`avatarUrl` on first sign-in. Once a user saves their profile manually, OAuth sync is disabled for that user (controlled by `profileUpdatedAt`).

## Step 10 — Verify the integration

Run your local dev server and perform the following checks:

| Check | Expected |
|---|---|
| `GET /admin` as non-admin user | Redirect to login or 403 |
| `GET /admin` as admin user | Admin dashboard renders |
| `GET /admin/users` | User list renders |
| `GET /admin/locks` | Locked/frozen account table renders (may be empty) |
| `GET /admin/waitlist` | Pending users table renders |
| `GET /admin/invites` | Invite code list renders; Create button visible |
| `GET /admin/api-keys` | API key list renders; Create button visible |
| `GET /admin/config` | Config table with source badges renders |
| Login with wrong password 3 times | Account frozen — 429 returned with `retryAfterSeconds` |
| Register with `requireInviteCode=true` and no code | `400` (generic registration error) |
| `Authorization: Bearer <validKey>` on protected route | 200 with `principal` data |
| `GET /api/account/profile` | Returns `{ displayName, avatarUrl, bio }` |

## Troubleshooting

**Admin routes return 404**
The route files in Step 4 were not created, or the file paths do not match. Verify `src/app/api/auth/admin/locks/route.ts` (etc.) exist and export the correct handlers.

**Admin page shows "Forbidden"**
The signed-in user does not have `role = "admin"` in the database. Run the bootstrap step (Step 7) or update the row directly.

**Lockout not triggering**
`accountLockout.enabled` is `false` in config. Check your env var and confirm the config object passed to `createSecureAuth` has `accountLockout.enabled: true`.

**Invite codes not enforced at registration**
Confirm `invites.requireInviteCode: true` in config. Registration must include `inviteCode` in the JSON body; OAuth self-registration is blocked when this flag is on. Pending accounts (`invites.requireApproval: true`) cannot sign in until an admin approves them via the waitlist.

**M2M key validation fails**
Ensure `apiKeys.enabled: true` in config. The `withApiKeyAuth` wrapper reads from `services.apiKeyService` — if the service is not wired (feature disabled), it throws. Confirm the key was created via the admin panel (raw key is shown once).

**Config overrides not applying**
In-memory cache TTL defaults to 60 seconds. Changes applied via `AdminConfigPage` take effect within the TTL window. Reduce `admin.configCacheTtlSeconds` for faster propagation during development.

## Security hardening upgrade (0.4.x+)

**Audience:** Consumers upgrading from `0.4.0` / `0.4.1` (or earlier) to a release that includes the security audit hardening.

Most protections are **automatic** once you upgrade and apply the config checklist below. No database migration is required for this release.

### What you get automatically after upgrade

| Area | Behavior |
| --- | --- |
| Admin API | Requires a **fully authenticated** session (2FA completed when enabled) |
| Account status | `users.status` (`active` / `pending` / `suspended`) enforced on every login path |
| Invites / waitlist | `requireInviteCode` and `requireApproval` enforced at registration and OAuth |
| 2FA enable | All existing sessions revoked; JWTs issued before 2FA activation invalidated |
| Admin mutations | Same-origin protection on POST/PATCH/DELETE admin routes |
| Sensitive overrides | HIBP, lockout, invite gates, magic link toggle, etc. **cannot** be weakened via admin panel |

### Step 1 — Upgrade the package

```bash
npm install @tgoliveira/secure-auth@latest
```

Rebuild your app so route handlers and UI pages from the package are picked up.

### Step 2 — Production config (required)

In production, in-memory rate limiting is **rejected at startup**. Add:

```typescript
createSecureAuth({
  // ...
  server: {
    environment: process.env.NODE_ENV === "production" ? "production" : "development",
    cookieSecure: process.env.NODE_ENV === "production",
  },
  rateLimit: {
    store: process.env.NODE_ENV === "production" ? "postgres" : "memory",
  },
});
```

| Setting | Production | Development |
| --- | --- | --- |
| `server.environment` | `"production"` | `"development"` or `"test"` |
| `rateLimit.store` | `"postgres"` (required) | `"memory"` (default) |

Ensure the `rate_limit` table exists (same Drizzle schema as the package). Multi-instance deployments **must** use Postgres rate limiting.

### Step 3 — Trusted proxy headers (if behind a load balancer)

Rate limiting no longer trusts `X-Forwarded-For` by default. If your app sits behind nginx, Cloudflare, or similar:

```typescript
security: {
  trustForwardedHeaders: true, // only when the edge proxy strips/forwards client IP correctly
},
```

Without this flag, per-IP rate limits use a single shared bucket (`unknown-ip`). Do **not** enable it on apps exposed directly to the internet without a trusted proxy.

### Step 4 — Magic link page (required if magic link is enabled)

Email links now point to a **UI page**, not `GET /api/auth/magic-link/verify`.

1. Add a Next.js page:

```typescript
// src/app/login/magic-link/page.tsx
export { MagicLinkVerifyPage as default } from "@tgoliveira/secure-auth/react";
```

2. Optional: customize the path in config:

```typescript
ui: {
  paths: {
    magicLinkVerify: "/login/magic-link", // default
  },
},
```

The page reads `?token=` client-side and POSTs to `/api/auth/magic-link/verify`. The API **no longer** exposes a GET handler with the token in the query string.

### Step 5 — Password reset UI (if you built a custom form)

The package removed the token validation oracle (`action: "validate"` on `POST /api/auth/reset-password`).

- **Package UI:** `ResetPasswordPage` shows the form when a token is present; validity is checked only on submit.
- **Custom UI:** Remove calls to `accountAuthApi.validateResetToken()`. Submit `{ token, newPassword }` directly to `POST /api/auth/reset-password`.

### Step 6 — Invites and waitlist (if enabled)

If you use `invites.requireInviteCode` or `invites.requireApproval`, enforcement is now live:

| Flag | Consumer action |
| --- | --- |
| `requireInviteCode: true` | Add an `inviteCode` field to your registration form; send it in `POST /api/auth/register` |
| `requireApproval: true` | New users get `status: "pending"`; show `WaitlistPendingPage` at `/waitlist` (or your `ui.paths.waitlistPending`) |
| OAuth + `requireInviteCode` | OAuth cannot create new accounts; users must register with a code first |

Admin approval flow is unchanged: `POST /api/auth/admin/waitlist` with `{ userId }`.

### Step 7 — Admin config panel expectations

These keys are **no longer overridable** at runtime (set via env / deployment only):

- `passwordPolicy.checkBreachedPasswords`, `passwordPolicy.enforcement`
- `accountLockout.enabled`
- `invites.requireApproval`, `invites.requireInviteCode`
- `auth.magicLink.enabled`, `auth.securityNotifications.enabled`
- `sessions.singleActiveSession`

Operational keys (profile toggle, invite quotas, `sessions.maxAgeSeconds`, etc.) remain overridable.

### Step 8 — Verify after upgrade

| Check | Expected |
| --- | --- |
| App boots in production with `rateLimit.store: "memory"` | Startup error mentioning postgres |
| Admin API with 2FA pending | `401` until TOTP completes |
| Suspended user login | Rejected (generic error) |
| Register without invite when `requireInviteCode` | `400` |
| Magic link email | Link opens `/login/magic-link?token=...` and completes sign-in |
| Enable 2FA on account with other sessions | Other browsers signed out |
| Admin POST without `Origin` header | `403` same-origin error |

### Breaking API summary

| Removed / changed | Migration |
| --- | --- |
| `GET /api/auth/magic-link/verify?token=` | Use `MagicLinkVerifyPage` + `POST` verify |
| `POST /api/auth/reset-password` with `action: "validate"` | Remove; submit reset directly |
| `accountAuthApi.validateResetToken()` | Removed from client SDK |
| `challengeToken` in login/passkey JSON when 2FA pending | Cookie only (no action needed if you already use cookies) |
| Admin override of security-sensitive keys | Use env / redeploy |
